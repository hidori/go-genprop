# 開発

- コンテキストウィンドウの制限を考慮し、要点を明確にする
- 明確な指示が無い限り、修正点のコミットを自発的に行う必要はない

## コード生成・修正

- 生成前に既存の linter エラーを把握する
- 生成後は複数の linter ルールを同時に満たす
- 記述量が減る、可読性が向上する、性能が向上する、などの明確なメリットのない提案をしない
- 共通化可能なコードは積極的に関数化する
- 明確な指示が無い限り、生成・修正コードに実装コメントを含めない
  - パブリックスコープの godoc コメントは必須とする
  - 既存のコメントは削除せずに維持する
  - 問題回避などのために補足が必要な場合のコメントは許容する
- コード内の空行は linter エラー（特に wsl）が発生しない範囲で最小化する
  - 関数内の記述で責務や意味の区切りとして空行を入れることは許容する
- 出来るだけ nolint コメントは使用しない
  - 使用する場合は nolint のスコープを最小限とし、理由を nolint コメントに付記する

## 一時ファイル

- テストやデバッグ用の一時ファイルは `tmp/` ディレクトリ配下に作成する
- `tmp/` ディレクトリは `.gitignore` に含まれており、Git管理対象外
- `make clean` により `tmp/` ディレクトリは削除される
