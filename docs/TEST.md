# テスト

## ファイル名

- テストファイルのファイル名の末尾は `_test.go` とする
- モックファイルのファイル名の末尾は `_mock.go` とする

## テスト関数

- 原則、テーブルドリブンテストを使用する
  - エッジケースのテスト等、テーブルドリブンテストの使用が妥当でない場合はこの限りではない
- テスト関数名は `Test` + 機能名とする

## テストケース

- テスト対象の関数が `err=nil` を返す場合のテストケースを「正常系のテスト」と呼ぶ
- テスト対象の関数が `err!=nil` を返す場合のテストケースを「異常系のテスト」と呼ぶ
- テストの `name` は英語で記述
- 正常系のテストの `name` の値の先頭に `success:` をつける
- 異常系のテストの `name` の値の先頭に `failure:` をつける

## テスト実行

- テストが実行できない等の有害な副作用が無い限り、テスト関数の冒頭と `t.Run()` 冒頭で `t.Parallel()` を使用する
- `testify/assert` を使用する
- `testify/require` を使用する
  - エラーを返す関数のテストでは `require.NoError()`, `require.Error()` を適切に使用する

## テストデータ

- レポジトリ直下の `testdata/` 配下のサブディレクトリに配置する
  - 単一パッケージ public/generator/ のテストで使用するテストデータは testdata/public/generator/ 配下に配置する
  - 複数パッケージのテストで使用するテストデータは、それらのパッケージが配置されたディレクトリパスの共通部分と対応する testdata/ 配下のサブディレクトリに配置する
    例： internal/app/formatter, internal/app/parser で共有されるテストデータは testdata/internal/app/ 配下に配置する
- testdata/ 配下のファイルがテストケースと1対1対応する時、以下の命名規則を適用する
  - ファイル名には、対応するテストケースが類推しやすい名前を使用する
    - ファイル拡張子は任意とする
  - ファイルがテストケースの入力として使用される時、ファイル名の末尾に `_input` をつける
  - ファイルがテストケースの出力検証に使用される時、ファイル名の末尾に `_output` をつける
- 出来る限り、ファイルの参照には go:embed を使用する
- go:embed を使わずにテストデータを読み込む場合、パス指定は文字列リテラルを使用する
